- - ### **大数据分析平台 - 最终简历呈现**

    

    **项目名称：大数据分析平台 (核心开发成员)**

    **项目周期:** 20xx年xx月 – 20xx年xx月

    **技术栈:** Spring Boot, Spring Cloud Alibaba, MyBatis-Plus, MySQL (元数据存储), Redis, Hive, ClickHouse, ANTLR, Docker

    **项目描述:** 一款企业级自助式商业智能（BI）平台，作为公司现有数据仓库之上的**智能查询与分析层**。平台旨在赋能无技术背景的业务与数据分析人员，允许他们通过拖拽式操作，对底层Hive与ClickHouse中的海量数据进行高效、灵活的自助分析。其核心在于，用户可以**自主创建并组合业务指标**，平台后端则负责将复杂的分析意图**自动翻译**为高性能的数据库查询。

    **职责与贡献:**

    - **主导设计并开发了“智能查询下推”引擎，实现了业务语言到SQL的精准翻译**
      - 构建了平台的**语义层**，支持用户在界面上创建并管理业务指标（如“利润率”），并实现了指标间的**嵌套引用**（如 `[利润率] = [利润] / [收入]`）。
      - 引入**ANTLR**技术，负责将用户输入的、包含嵌套引用的复杂公式字符串，安全地解析为抽象语法树（AST），实现了对用户分析意图的深度校验与结构化理解。
      - 基于**构建者（Builder）\**与\**策略（Strategy）\**模式，开发了一个强大的SQL构建器。它能根据AST和目标数据源（Hive/ClickHouse），智能规划并生成包含\**多层CTE、窗口函数、动态透视**等复杂逻辑的高性能SQL。
    - **设计并实现了多级缓存系统，显著提升了前端交互性能**
      - 为应对高并发和重复查询，主导设计了基于**Redis**的结果集缓存方案。
      - 独立完成了缓存Key的规范化设计（排序与哈希）、TTL更新策略，并运用分布式锁等机制解决了缓存雪崩、穿透等高并发场景下的稳定性问题。
      - 该系统使高频访问仪表盘的响应时间从**秒级降至毫秒级**，并将底层数据仓库的重复计算负载**降低了60%以上**。
    - **开发了跨源数据融合模块，支持高级混合分析**
      - 实现了平台的**跨源查询**能力。当一个分析请求同时依赖Hive和ClickHouse的指标时，查询引擎会自动将任务分解。
      - 通过并发下推将子查询分发至各数据源，在获取各自的中间结果后，由**Java后端在内存中进行最终的数据融合与计算**，为用户提供了统一、无缝的分析体验。
    - **负责设计高可用的数据查询API，保障平台整体稳定性**
      - 为前端仪表盘提供了稳定、低延迟的数据查询接口。
      - 在“一图一查”的并发模型下，通过配置独立的**有界线程池**，对数据查询任务进行了有效的**资源隔离**，成功防止了偶发的Hive慢查询拖垮整个应用，确保了平台在高负载下的高可用性。







**项目名称：** 企业级一站式自助BI数据分析平台 **项目角色：** Java后端核心开发 **技术架构：** Spring Boot, Spring Cloud Alibaba, **Apache Calcite (SQL解析)**, MySQL, Redis, XXL-JOB, EasyExcel **项目描述：** 针对业务部门高度依赖IT人工提数痛点，构建的一站式自助BI平台。平台支持异构多数据源（MySQL, Oracle, Hive等）的动态接入，通过“拖拽式”交互自动生成报表。采用 **Strategy策略模式** 兼容多种数据库方言，目前支撑日均活跃用户2000+，作为核心报表平台承载了全公司的日常数据分析与定时推送业务。

**核心职责与技术产出：**

- **自主研发OLAP多维分析语义层（核心难点）：**

  - **SQL编译与生成：** 基于 **Apache Calcite** 构建语义转换引擎，将前端拖拽的JSON元数据（维度、指标、筛选、聚合）解析为 **AST（抽象语法树）**。
  - **异构方言适配：** 实现了通用的SQL生成器，能够动态生成适配 MySQL、Oracle、ClickHouse 等不同数据库语法的执行SQL，**100%屏蔽底层数据库差异**，实现了“一次拖拽，多源运行”。
  - **性能优化：** 针对复杂报表解析慢的问题，设计了基于 Redis 的元数据缓存策略，将核心解析链路的耗时从 **200ms降低至50ms以内**。

- **高并发查询的资源隔离与熔断（舱壁模式）：**

  - **查询并发控制：** 针对“单报表多图表”场景引发的 **请求放大效应**（如打开一张报表触发10+次API调用），摒弃了直接在Web线程执行SQL的传统方式，引入**全局查询线程池**对数据库查询任务进行统一托管。通过严格限制核心线程数，有效防止了瞬间爆发的查询流量打垮数据库连接池。
  - **分级舱壁隔离：** 设计了基于业务优先级的**双路线程池隔离策略**。为“核心管理驾驶舱”配置独立的 **VIP查询队列**，与普通报表查询进行物理隔离。即使普通业务出现因复杂SQL导致的线程阻塞（排队），核心业务的报表加载依然能保持毫秒级响应，实现了**故障级联阻断**。

- **高稳定性的大数据流式导出（OOM优化）：**

  - **流式处理：** 针对业务导出百万行级数据的需求，重构了原有内存加载模式，采用 **EasyExcel + 数据库流式游标（Cursor/Pagination）** 策略，实现数据的边读边写，内存占用恒定。
  - **资源保护：** 引入**背压机制**与异步线程池，通过限制并发导出任务数，成功避免了大数据量导出时的 **Full GC** 和 **OOM** 风险，确保在导出高峰期系统依然稳定。
